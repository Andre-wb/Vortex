# Blind Mailbox Protocol (BMP) — Исследование

> Приватность метаданных в мессенджере без Tor, mixnet и PIR.
> Разработано для Vortex Messenger.

---

## 1. Постановка проблемы

Все существующие мессенджеры (Signal, Telegram, WhatsApp, Vortex) имеют одну нерешённую проблему: **сервер знает кто кому пишет**. Содержимое сообщений зашифровано (E2E), но сам факт коммуникации — нет.

Формально: сервер наблюдает `(IP_отправителя, ID_получателя, время, размер)` для каждого сообщения с 100% достоверностью.

### Существующие решения и их фатальные недостатки

| Подход | Задержка | Трафик | Мобильные | Проблема |
|--------|----------|--------|-----------|----------|
| Tor/Onion routing | 300-500мс | 3x | Плохо | Слишком медленно для чата |
| Mixnet (Nym, Katzenpost) | 5-30 секунд | 2x | Нет | Realtime невозможен |
| PIR (Private Information Retrieval) | 50мс | 100-1000x | Нет | Гигабайты трафика |
| Signal Sealed Sender | 50мс | 1x | Да | Сервер всё равно видит получателя |

**Все существующие подходы либо убивают UX, либо не решают проблему полностью.**

## 2. Blind Mailbox Protocol — концепция

### Идея

Сервер не маршрутизирует сообщения по `user_id`. Вместо этого:

- Каждая пара пользователей вычисляет **общий "почтовый ящик"** из shared secret
- Ящик имеет случайный ID, **меняющийся каждый час**
- Сервер **не знает** какой ящик принадлежит какому пользователю
- Cover traffic делает реальные операции неотличимыми от фейковых

### Математика

При первом контакте Алиса и Боб вычисляют общий секрет через ECDH:

```
S_AB = ECDH(Alice_private, Bob_public) = ECDH(Bob_private, Alice_public)
```

Mailbox ID вычисляется детерминированно:

```
mailbox_id(t) = HMAC-SHA256(S_AB, floor(t / T))[0:16]

где:
  T = период ротации (3600 секунд = 1 час)
  floor(t / T) = номер текущей эпохи
  [0:16] = первые 16 байт (128-битный ID)
```

Оба знают `S_AB` -> оба вычисляют **одинаковый** mailbox_id в любой момент времени.

### Протокол отправки

```
Алиса -> серверу:
  POST /mailbox/a7f3b2c8e1d4...
  Body: AES-256-GCM(сообщение, room_key)

Сервер:
  Сохраняет в таблицу: {mailbox_id -> [сообщения]}
  НЕ знает кто Алиса и кто получатель
```

### Протокол получения

```
Боб каждые 2-3 секунды:
  1. Вычисляет свои реальные mailbox ID (по одному на каждый контакт)
  2. Генерирует 50 фейковых случайных mailbox ID
  3. Перемешивает все в случайном порядке
  4. Запрашивает каждый: GET /mailbox/{id}
  5. Реальные ответы -> расшифровывает
  6. Фейковые ответы -> отбрасывает

Сервер видит 60 запросов к 60 разным ящикам.
Не может определить какие 10 из 60 — реальные.
```

## 3. Формальный анализ безопасности

### Модель угрозы

Противник = сервер (полный контроль над инфраструктурой):
- Видит все HTTP запросы (IP, URL, время, размер)
- Может хранить историю неограниченно
- Может проводить статистический анализ
- **Не может** модифицировать данные (целостность через E2E)

### Атака 1: Корреляция по времени

**Вектор**: Алиса пишет в `mb_x` в 19:23:05, Боб читает `mb_x` в 19:23:07. Сервер связывает.

**Защита**: Пакетный polling — Боб читает ВСЕ свои ящики (реальные + фейковые) каждые `T_poll` секунд, независимо от наличия новых сообщений. Время чтения не коррелирует со временем записи.

**Остаточная утечка**: при `T_poll = 3с` сервер знает что сообщение прочитано в окне 3 секунд. Но не знает КЕМ (60 клиентов могли прочитать этот ящик в это окно).

### Атака 2: Анализ объёмов

**Вектор**: Ящик `mb_x` получает 20 сообщений/час. Следующий час `mb_y` тоже получает 20/час с похожим паттерном.

**Защита**: Фейковые записи — каждый клиент пишет в 5-10 случайных ящиков каждый час (зашифрованный мусор). Сервер видит что большинство ящиков получают 1-3 сообщения/час, не может отличить реальные от фейковых.

**Формально**: Пусть C = cover ratio (фейковые/реальные). Вероятность правильной идентификации реального ящика:

```
P(верно) = 1 / (1 + C)

При C = 5:  P = 1/6  = 16.7%
При C = 10: P = 1/11 = 9.1%
При C = 20: P = 1/21 = 4.8%
```

### Атака 3: Корреляция по IP

**Вектор**: IP_1 пишет в `mb_x`, IP_2 читает `mb_x`. Значит IP_1 общается с IP_2.

**Защита**:
1. Cover traffic — IP_2 читает 50+ ящиков, не только `mb_x`
2. Федерация — запись и чтение через разные ноды
3. Опционально: Tor/VPN для транспортного уровня

**С федерацией**: Нода_A хранит ящики `0-7*`, Нода_B хранит `8-F*`. Алиса (на Нода_A) пишет в ящик `a3f2...` -> остаётся на Нода_A. Боб (на Нода_B) читает `a3f2...` с Нода_A через inter-node API. Нода_A не знает что Боб на Нода_B. Нода_B не знает что Алиса писала в `a3f2...`.

### Атака 4: Связывание при ротации

**Вектор**: В 20:00 `mb_old` перестаёт получать сообщения, `mb_new` начинает. Сервер связывает.

**Защита**: Период перекрытия — оба ящика активны 10 минут. Плюс per-pair jitter:

```
время_ротации(пара) = T + HMAC(S_AB, "jitter")[0] mod 600
```

Каждая пара ротируется в разное время (+-10 минут). Сервер не может сопоставить.

### Атака 5: Долгосрочный статистический анализ

**Вектор**: За месяцы сервер накапливает паттерны. Реальные ящики имеют чуть более регулярные чтения чем фейковые.

**Защита**: Равномерный polling — клиент читает ВСЕ ящики (реальные и cover) с идентичной частотой и паттерном. Cover ящики не просто генерируются — они "живут" неделями (клиент помнит свои cover ID и продолжает их опрашивать).

**Формальная гарантия**: Если клиент опрашивает `N_real + N_cover` ящиков с равной частотой, advantage различения:

```
Adv <= N_real / (N_real + N_cover) + negl(lambda)

Для 10 реальных + 50 cover: Adv <= 10/60 = 16.7%
```

## 4. Открытые проблемы

### Проблема 1: Групповые чаты

В группе из N человек — отправитель должен записать в N-1 ящиков (по одному на каждого участника). Для группы из 100 -> 99 записей на одно сообщение.

**Возможное решение**: Древовидная рассылка — отправитель пишет в log2(N) ящиков, каждый промежуточный участник пересылает дальше. Задержка увеличивается на `hop * T_poll`, но записей = O(log N) вместо O(N).

**Альтернатива**: Один общий ящик для группы + per-member суб-ящики для доставки. Компромисс: сервер видит что группа из ~N человек существует, но не знает кто в ней.

### Проблема 2: Push-уведомления

Сервер должен знать кому слать push. Но push = раскрытие "этот пользователь ждёт сообщение".

**Возможное решение**: Push Proxy — отдельный сервис, не связанный с mailbox сервером. Клиент регистрирует `push_token` на proxy. Mailbox сервер отправляет анонимный сигнал proxy: "отправь push всем подписчикам категории X". Категория = хеш от mailbox_id. Proxy не знает содержимое, mailbox сервер не знает push_token.

### Проблема 3: Первый контакт

Как Алиса узнаёт mailbox_id Боба если они ещё не обменялись ключами?

**Решение**: Публичный "intro mailbox" привязанный к хешу публичного ключа:

```
intro_mailbox(Bob) = SHA256(Bob_public_key)[0:16]
```

Это **одноразовый** ящик. Алиса записывает свой ECDH ephemeral key. Боб проверяет периодически. После обмена -> переходят на ротируемые ящики. Intro mailbox больше не используется.

Утечка: сервер видит что кто-то писал в intro_mailbox Боба. Но не знает кто (если Алиса использует cover traffic для intro).

### Проблема 4: Голосовые/видео звонки

Realtime медиа требует постоянный двунаправленный поток. Blind Mailbox не подходит (2-3 секунды задержки для polling).

**Решение**: Для звонков использовать отдельный механизм — onion-routed media через SFU. Сигнализация (SDP offer/answer) идёт через Blind Mailbox (2-3 секунды задержки допустимы для setup). Медиа идёт через зашифрованный relay с маскировкой IP.

### Проблема 5: Несколько устройств

Один пользователь на 3 устройствах — все 3 должны читать одни и те же ящики. Сервер видит 3 IP читающих одни и те же 10 ящиков -> связывание.

**Решение**: Каждое устройство имеет уникальный набор cover ящиков. Пересечение только на реальных (10 из 60 совпадают). С 50 cover — 10/60 = 16.7% overlap, статистически неотличимо от случайного совпадения.

## 5. Пропускная способность и задержка — финальные цифры

```
Параметры:
  Активных переписок: 10
  Cover ratio: 5x (50 фейковых ящиков)
  Интервал polling: 3 секунды
  Размер запроса: ~100 байт
  Размер ответа: ~200 байт (обычно пустой)
  Период ротации: 1 час

Наивный подход (отдельные запросы):
  Запросов в час = 3600/3 = 1200
  Ящиков за запрос = 60
  Всего запросов/час = 72,000
  Всего байт/час = 72,000 * 300 = 21.6 МБ/час  (слишком много)

Оптимизированный (пакетные запросы):
  GET /mailbox/batch?ids=a7f3,b2c8,d4e5,...
  Один HTTP запрос вместо 60.
  
  Всего запросов/час = 1200 (пакетных)
  Размер пакета = 60 * 32 = 1920 байт запрос + ~500 байт ответ
  Итого = 1200 * 2420 = 2.9 МБ/час = 48 КБ/мин

  48 КБ/мин — полностью приемлемо. Меньше чем фоновое обновление приложений.

Задержка:
  Средняя: T_poll / 2 = 1.5 секунды
  Худший случай: T_poll = 3 секунды
  
  Для сравнения:
    Email:          минуты-часы
    SMS:            1-5 секунд
    Telegram:       50-200мс
    Blind Mailbox:  1.5 секунды (средняя)
```

## 6. Что знает сервер — сравнение

```
                          Текущий Vortex    С Blind Mailbox
Кто отправил:             100% известно     ~17% угадывание (при C=5)
Кому:                     100% известно     ~17% угадывание
Когда:                    точно до мс       окно +-3 сек
Размер:                   точно             точно (нельзя скрыть без padding)
Содержимое:               0% (E2E)          0% (E2E)
Частота общения:          точно             ~17% угадывание
Граф контактов:           полный            неизвестен серверу
```

## 7. Оценка реализации для Vortex

### Что нужно:
1. Новые endpoints: `POST /mailbox/{id}`, `GET /mailbox/batch`
2. Клиентский модуль: вычисление mailbox ID, cover traffic, polling
3. In-memory хранилище для ящиков (Redis для production)
4. Миграция: существующие чаты -> blind mailbox (переходный период)

### Оценка трудозатрат:
- Backend: ~500 строк Python
- Frontend: ~400 строк JavaScript
- Тесты: ~200 строк
- Общее время: 2-3 недели до рабочего прототипа

### Совместимость:
Работает поверх существующей E2E инфраструктуры. Room keys, ECIES, ratchet — всё остаётся. Меняется только **транспортный слой**.

## 8. Сравнение с существующими разработками

| Свойство | Signal Sealed Sender | Tor | Mixnet | Pond | Briar | **Blind Mailbox** |
|----------|---------------------|-----|--------|------|-------|-------------------|
| Отправитель скрыт от сервера | Частично | Да | Да | Да | Да | **Да** |
| Получатель скрыт от сервера | Нет | Да | Да | Да | Да | **Да** |
| Задержка | 50мс | 300мс+ | 5-30с | 30с+ | 500мс+ | **1.5с средняя** |
| Overhead трафика | 0% | 3x | 2x | 2x | 3x | **48 КБ/мин** |
| Работает на мобильных | Да | Еле-еле | Нет | Нет | Еле-еле | **Да** |
| Совместим с федерацией | Нет | Да | Частично | Нет | Нет | **Да** |
| Не требует Tor | Да | Нет | Нет | Нет | Нет | **Да** |
| Групповые чаты | Да | Нет | Нет | Нет | Частично | **O(N) или O(log N)** |
| Активный проект | Да | Да | Частично | Заброшен | Да | **Предложение** |

Ближайшие аналоги:
- **Pond** (Адам Лэнгли, 2013) — использовал Tor + mixnet для скрытия метаданных, заброшен
- **Briar** — использует Tor hidden services для P2P, неэффективен для нескольких устройств
- **Cwtch** — похож на Briar с поддержкой групп, требует Tor

Ни один из них не использует ротируемые хеш-производные ID ящиков с cover traffic в федеративной сети без зависимости от Tor.

## 9. Заключение

Blind Mailbox Protocol — не теоретическая абстракция, а **практически реализуемый** протокол с конкретными цифрами:

- **48 КБ/мин** overhead (приемлемо)
- **1.5 сек** средняя задержка (приемлемо для чата, не для звонков)
- **83%** сокращение серверного знания о метаданных
- **Совместим** с федерацией, E2E, существующей кодовой базой
- **Не требует Tor/VPN** (cover traffic обеспечивает сравнимые гарантии)

Это может быть **первый мессенджер** где сервер доказуемо не знает граф контактов своих пользователей — без ущерба для мобильного UX.
