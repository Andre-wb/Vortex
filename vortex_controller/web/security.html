VORTEX / SECURITY

Security model

Exactly what's cryptographically verified, what's not, and how each key in the system earns — or loses — your trust. Read this before you connect new clients to a controller you don't run yourself.

1. Keys in play

Four independent Ed25519/X25519 keypairs, each with a single, narrow job. Loss or compromise of any one key does not break the others.

Key Lives Signs Verified by
Release key Developer's HSM / air-gap Source code manifest (INTEGRITY.sig.json) Clients — pubkey pinned at build time
Controller key VPS (keys/controller.key) Every /v1/* response payload Clients — pubkey read from SNS or build pin
Node signing key Each Vortex node's keys/ dir Its own registration & heartbeats Controller — verifies against pubkey in payload
User identity (X25519) Mobile / desktop device only Session handoff tokens, message envelopes Receiving node — never leaves the device

2. Trust flow when a client connects

Each step validates the next. If any check fails the client refuses the connection and falls back to the next controller URL.

1
Resolve via SNS
Read vortexx.sol on-chain: URL + TXT records. The record is signed by the Solana wallet that owns the name — impossible to DNS-hijack.
2
Integrity probe
Client calls /v1/integrity. Rejects anything other than status: verified and a signed_by matching the pinned release key.
3
Signed responses
Every subsequent API call returns {payload, signature, signed_by}. Client verifies the Ed25519 signature with the controller pubkey from step 2.
4
Cross-check on Solana
For sealed nodes, client optionally re-reads the code_hash from the on-chain registry. Divergence from the controller's self-reported hash = tampered — node permanently blacklisted.

3. What is actually protected

Green = cryptographically enforced. Red = trust assumption or out-of-scope.

✓
Response authenticity — impossible to forge /v1/nodes/random without the controller's private key. A MITM can't swap in malicious peer URLs.
✓
Code integrity — the running controller's files are hashed and cross-checked against a signed manifest at every startup. Runtime tampering fails the gate middleware and returns 503.
✓
Node identity — a node must sign its registration with its own private key. An attacker can't register on behalf of another node's pubkey.
✓
Handoff / migration — every session transfer carries an Ed25519-signed token with anti-replay, valid only for 5 minutes. Target node rejects unknown source nodes.
✓
DNS hijack — .sol records are on Solana, not DNS. An adversary needs to compromise the wallet that owns the domain — not just a DNS registrar.
✗
Transport privacy — the URL you connect to is visible to your ISP/Cloudflare. Use Tor mirror if metadata matters. End-to-end message content is encrypted separately.
✗
Message content — is NOT signed by the controller or nodes. It's encrypted end-to-end between users. Controller can't read anything even if compromised.
✗
Availability — a DDoS or domain seizure can take the controller offline. Fallback = mirrors + multiple controllers + on-chain Solana registry.
✗
Host OS / kernel — if the VPS itself is rooted, keys can be stolen. This attestation layer does not protect against it. Use hardware-backed signing for paranoid deployments.

4. What the badges on /nodes mean

Pick the verification level that fits your threat model. Any node can claim anything; these badges only confirm what's cryptographically supported.

✓✓ DUAL Two witnesses

Registered with the controller AND sealed on-chain. Both the HTTP signature and the Solana record agree on the node's pubkey + code hash. Safe for long-lived sessions with sensitive data.

✓ CTRL Controller only

Proves key ownership (signed registration) but no on-chain seal. Fine for short sessions; risk model is "trust the controller operator". If controller is compromised, this node could be fake.

⚠ UNVERIF Bootstrap

Known from a config file or gossip message. No cryptographic proof yet. Use only to bootstrap the peer list, then replace with verified entries. Avoid for session handoff targets.

6. If something looks wrong

Red flags and the corresponding action.

!
Controller pubkey on this page differs from the one pinned in my app — stop connecting. Either your app is out of date, or this site is fake. Compare with the pubkey shown on the Vortex release page.
!
Integrity badge on /v1/integrity says "tampered" — the controller itself won't serve traffic (gate middleware blocks it). Apps automatically switch to a fallback controller.
!
A node's code_hash on /nodes doesn't match what it reports — treat as compromised. Clients mark it permanently and refuse it as a handoff target.
!
This whole page is unreachable from your ISP — not a security issue by itself. Use a mirror from the Mirrors page.
